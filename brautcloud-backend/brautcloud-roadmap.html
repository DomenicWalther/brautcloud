The Roadmap

5 Weeks · 6 Phases

Each week maps directly to the phases you're building that week — backend and frontend together.

Week 1
Foundation
Wire up the HTTP layer & core services — nothing else works without this.
—
Frontend Infrastructure Setup
FE Only
Before any API integration is possible, the Angular app needs its HTTP layer, services, and auth interceptor configured. This is the prerequisite for every other week.
Add provideHttpClient(withInterceptors([authInterceptor])) to app.config.ts
Create environment.ts with apiUrl and frontendUrl
Build Auth Interceptor — auto-injects JWT from localStorage or sessionStorage
Create EventService, ImageService, AuthService skeletons
Set up Auth Store using Angular Signals (not NgRx — overkill for MVP)
Build reusable components: Button, Input, Modal, Toast notification service
01
QR Code Generation
Good First Win
Couples get a scannable QR code when they create an event. Satisfying, visible progress that unblocks the entire guest flow.
Backend
01Add ZXing core + javase to pom.xml
02Create QRCodeService.generateQRCodeImage(url) → byte[], 300×300, error correction M
03Endpoint: GET /api/events/{id}/qrcode → image/png, generated on-demand
Frontend
01Add getQRCode(id) to EventService
02Replace placehold.co image in Home dashboard with real API call
03Download button (canvas blob) + copy link button
Week 2
Auth & Image Viewing
Wire up login for couples and unlock the ability to actually see photos.
—
Connect Couple Authentication
FE Only
Backend auth endpoints already exist — this week is about wiring the Angular forms to them.
Wire sign-in form → POST /api/auth/login, store JWT in localStorage
Wire sign-up form → POST /api/auth/register
Add route guards protecting all /app/* routes
Handle token refresh + logout flow end-to-end
04
Presigned URLs for Image Viewing
Critical — Unblocks Everything
Temporary S3 access links so images can be displayed without exposing the bucket. Nothing can be shown to guests or couples without this.
Backend
01Add getPresignedUrl(key, minutes) to S3Service — expiry: 15 min
02Add presignedUrl + urlExpiresAt fields to ImageResponse DTO
03Endpoint: GET /api/image/event/{id}/all (couple token, all images)
04Endpoint: GET /api/image/event/{id}/public (guest token, visible only)
Frontend
01Connect ImageGallery to real API — replace 12 placeholders
02Use presignedUrl directly in <img> src
03Auto-refresh image list every 10 min (URLs expire at 15)
04Handle 403s on expired URLs → refetch list automatically
Note

S3 bucket must be fully private. Generate URLs fresh per request — no server-side caching. Keep server clock NTP-synced or presigned URLs will fail.

Week 3
Public Access & Guest Auth
Guests can now reach the event page and authenticate with a password.
02
Public Event Access & Guest JWT
Guests see basic event info without an account, enter the event password, and receive a short-lived guest JWT (4 hours). This enables the entire guest workflow.
Backend
01Open GET /events/{id} + POST /events/{id}/verify in Spring Security (no auth required)
02Public GET returns: name, location, date — no password fields ever
03Verify returns guest JWT with claims: type:guest, eventId, 4hr expiry
04Extend JwtAuthFilter to handle both user and guest token types
05Rate-limit /verify: max 5 req/min/IP to prevent brute force
Frontend (new routes)
01Create /event/:id — EventAccessComponent fetches public event info
02Create /event/:id/password — PasswordEntryComponent with error handling
03On success: store guest JWT in sessionStorage (not localStorage)
04Add guest route guards — redirect to password page if no session token
Decision

Single JWT secret with type=guest claim to differentiate token types. Simpler than dual secrets for MVP — extend later if needed.

Week 4
Guest Upload & Deletion
The core guest experience — anonymous upload and session-based deletion.
03
Anonymous Image Upload
Guests upload photos without creating an account. Their identity lives entirely in the JWT — no database profile needed.
Backend
01Add guestToken (String) + isGuestUpload (boolean) to Image entity
02Endpoint: POST /api/image/guest-upload — multipart, guest JWT required
03S3 key pattern: {guest-uuid}/{filename} — organises uploads by uploader
04Default isVisible=false — couple must approve before guests see it
Frontend
01Create /event/:id/upload — drag-drop + file picker
02Client-side validation: JPEG/PNG/WebP only, max 10MB
03Show preview thumbnails before upload (FileReader API)
04Upload progress bar via XHR progress events
05
Guest Session-Based Deletion
Frontend Heavy
Guests can delete their own uploads while the tab is open. sessionStorage clears on close — that's intentional.
Backend
01Endpoint: DELETE /api/image/guest/{id} → 204 No Content
02Verify image.guestToken === JWT UUID claim — else 403
03Delete S3 object first, then DB record (hard delete for MVP)
Frontend
01Track uploadedImageIds[] in component state after each upload
02Show "Your Uploads" section with delete buttons (only for current session's images)
03Optimistic UI — remove immediately from view, rollback on error
04Confirmation dialog before deleting
Week 5
Couple Dashboard & Polish
Full couple controls + the finishing touches that make it production-ready.
06
Couple Dashboard Controls
Couples manage their event fully — approve photos, hide content, delete anything, edit event settings.
Backend
01Endpoint: GET /api/events/{id} — authenticated couple, full detail
02Endpoint: PATCH /api/events/{id} — update name, date, location, password
03Endpoint: PATCH /api/image/{id}/visibility — toggle isVisible
04Endpoint: DELETE /api/image/{id} — couple deletes any image in their event
Frontend
01Full gallery management — all images, greyed-out styling for hidden ones
02Visibility toggle switches + filter: All | Visible | Hidden | Pending
03Event settings page: edit name, date, password
04Replace mock stats (1,243 / 340 / 3.8k) with real API data
—
Polish & Production Readiness
The details that turn a working app into a shippable one.
Toast notifications for all success and error states
Skeleton loaders for async states (gallery, dashboard)
Mobile responsiveness pass — camera upload on mobile
Unit + integration tests (Vitest + Testcontainers)
Audit all env vars — zero hardcoded secrets in both repos
CORS locked to frontend domain, HTTPS in production

Work Strategy

Event management → needs CRUD endpoints
Image display → needs presigned URLs
Guest flow → needs guest auth system
QR code → needs generation endpoint
🔀 Meet in the Middle

FE uses mock services while BE builds
Integration testing to wire together
